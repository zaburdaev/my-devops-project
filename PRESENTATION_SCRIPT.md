### **СЛАЙД 5: CI/CD Pipeline** ⏱️ 2 минуты

#### Что показано:
- Схема GitHub Actions pipeline
- Этапы CI/CD
- Скриншот успешного workflow

#### Что говорить:
> "Один из ключевых элементов DevOps - это **автоматизация**. Я настроил полноценный CI/CD pipeline с помощью **GitHub Actions**.
> 
> **Вот как это работает:**
> 
> **Триггер:**  
> Когда я делаю `git push` в ветку `main`, автоматически запускается pipeline.
> 
> **Этап 1: Continuous Integration (CI)**
> 1. **Checkout кода** - GitHub Actions клонирует репозиторий
> 2. **Запуск тестов** - выполняются все 12 unit тестов с помощью pytest
> 3. **Проверка качества кода** - линтеры проверяют код style (flake8, pylint)
> 4. **Сборка Docker образа** - если тесты прошли, собирается новый образ
> 5. **Push в Docker Hub** - образ загружается в registry с тегом версии
> 
> **Этап 2: Continuous Deployment (CD)**
> 6. **SSH подключение к AWS серверу** - через защищенные secrets
> 7. **Pull нового образа** - сервер скачивает свежий образ из Docker Hub
> 8. **Обновление сервисов** - docker-compose выполняет rolling update
> 9. **Health check** - проверка, что приложение поднялось корректно
> 10. **Уведомление** - в случае успеха или ошибки (можно добавить Slack/Email)
> 
> **Результат:**  
> От коммита до production развертывания проходит **около 5 минут**. Полностью автоматически. Без ручного вмешательства.
> 
> Все успешные запуски можно увидеть в GitHub Actions - зеленые галочки подтверждают, что код рабочий."

#### Ключевые моменты:
- **Zero-downtime deployment** - пользователи не замечают обновление
- **Автоматические rollback** - если тесты не прошли, deploy не происходит
- **Версионирование** - каждый образ имеет тег с commit hash
- **Безопасность** - SSH ключи и пароли хранятся в GitHub Secrets

#### Технические детали (если спросят):
- GitHub Actions workflow файл: `.github/workflows/ci-cd.yml`
- Используется self-hosted runner или GitHub-hosted
- Кэширование зависимостей для ускорения
- Matrix builds для тестирования на разных версиях Python
- Можно добавить staging окружение перед production

> **Вызов:** Тесты проходили локально, но падали в GitHub Actions.  
> **Решение:** Обнаружил, что не хватало переменных окружения. Добавил `.env.test` файл и настроил GitHub Secrets. Научился важности идентичности окружений.
> 
> **Проблема 3: Высокое потребление памяти на AWS**  
> **Вызов:** Сервер часто исчерпывал память, приложение падало.  
> **Решение:** Добавил resource limits в docker-compose, настроил swap, оптимизировал Gunicorn workers. Научился важности мониторинга - Grafana показала проблему.

> **Проблема 4: Долгий deploy**  
> **Вызов:** Каждый deploy занимал 10+ минут из-за пересборки всех образов.

> 
> **Проблема 5: Потеря данных при перезапуске контейнеров**  
> **Вызов:** После `docker-compose down` терялись данные БД.  
> **Решение:** Настроил Docker volumes для персистентности. Теперь данные сохраняются независимо от жизненного цикла контейнеров.
> 
> **Главный урок:**

## ❓ 4. ОТВЕТЫ НА ВОЗМОЖНЫЕ ВОПРОСЫ

### **Q1: Почему выбрали именно такую архитектуру?**

**Ответ:**
> "Я выбрал микросервисную архитектуру с разделением на отдельные компоненты по нескольким причинам:
> 
> 1. **Масштабируемость** - каждый сервис можно масштабировать независимо. Если нагрузка на приложение растёт, я могу поднять больше реплик Flask, не трогая базу данных.

> 2. **Изоляция** - сбой в одном контейнере не уронит всю систему. PostgreSQL работает независимо от Redis.

> 3. **Flexibility** - легко заменить компонент. Например, если захочу сменить Redis на Memcached, это не затронет остальные части.
> 
> 4. **DevOps best practices** - эта архитектура отражает то, как строят системы в реальных компаниях. Монолит проще для маленького проекта, но я хотел освоить production подходы.
> 
> 5. **Мониторинг** - с отдельными сервисами проще отслеживать, где проблема. Grafana показывает метрики каждого компонента отдельно."

### **Q2: Как именно работает ваш CI/CD pipeline? Опишите подробнее.**

**Ответ:**
> "Конечно! Вот детальный flow:
> 
> **Триггер:** Я делаю `git push` в ветку main.
> 
> **GitHub Actions получает событие** и запускает workflow, описанный в `.github/workflows/ci-cd.yml`.
> 
> **Job 1: Test & Build (CI)**
> - Runner создаёт чистое Ubuntu окружение
> - Клонирует код из репозитория
> - Устанавливает Python и зависимости
> - Запускает pytest - все 12 тестов должны пройти

> - Логинится в Docker Hub используя secrets
> - Пушит образ в registry

> **Job 2: Deploy (CD)**
> - Подключается к AWS EC2 серверу по SSH (ключ хранится в GitHub Secrets)

> - Старые контейнеры gracefully останавливаются
> - Новые поднимаются - zero downtime благодаря rolling update
> - Health check - проверка, что приложение отвечает на /health endpoint
> - Если health check failed - автоматический rollback к предыдущей версии
> 
> **Результат:** От коммита до production - полностью автоматизировано."

### **Q4: Как можно масштабировать ваше приложение при росте нагрузки?**

**Ответ:**
> "Архитектура спроектирована с учётом масштабирования. Вот варианты:
> 
> **Вертикальное масштабирование (Scale Up):**
> - Увеличить размер EC2 instance (с t2.medium на t2.large)
> - Одно изменение в Terraform: `instance_type = "t2.large"`
> - Просто, но есть лимит
> 
> **Горизонтальное масштабирование (Scale Out) - лучший подход:**
> 
> **1. Application tier:**
> - Поднять несколько реплик Flask приложения
> - В Docker Compose: `docker-compose up --scale web=3` - 3 экземпляра приложения
> - В Kubernetes: `kubectl scale deployment flask-app --replicas=5`
> - Добавить Load Balancer (AWS ALB) перед ними
> - Stateless приложение позволяет это легко сделать
> 
> **2. Database tier:**
> - PostgreSQL master-slave репликация
> - Read queries идут на slaves, write на master
> - Или перейти на AWS RDS с Multi-AZ
> - Connection pooling (PgBouncer) для эффективного использования соединений
> 
> **3. Caching tier:**
> - Redis cluster с несколькими нодами
> - Redis Sentinel для high availability
> - Или AWS ElastiCache
> 
> **4. Auto-scaling:**
> - В Kubernetes настроить Horizontal Pod Autoscaler (HPA)
> - Метрика: если CPU > 70%, добавить replica
> - Автоматически адаптируется к нагрузке
> 
> **5. CDN:**
> - Статические файлы (CSS, JS, images) через CloudFront
> - Снижает нагрузку на сервер
> 
> **Bottlenecks и оптимизация:**
> - Grafana покажет, где bottleneck - CPU, память, database?
> - На основе метрик решаем, что масштабировать
> 
> Kubernetes делает всё это тривиальным - меняешь число реплик в манифесте, остальное автоматически."

### **Q5: Какие именно метрики вы собираете и почему они важны?**

**Ответ:**
> "Я собираю метрики на трёх уровнях:

> - **Response time** - latency каждого endpoint. Если >500ms, нужна оптимизация
> - **Error rate** - процент 4xx и 5xx ошибок. Spike означает проблему
> - **Requests by endpoint** - какие URL наиболее популярны
> - **Active users/sessions** - сколько пользователей сейчас онлайн

> - **Query performance** - slow queries detection
> - **Database size** - рост данных
> - **Locks** - блокировки таблиц
> 
> **Почему это важно:**
> - **Проактивность:** вижу проблемы ДО того, как пользователи пожалуются
> - **Optimization:** метрики показывают, где узкие места
> - **Capacity planning:** тренды помогают планировать апгрейды
> - **Debugging:** когда что-то сломалось, метрики показывают причину
> - **SLA tracking:** могу измерить uptime и availability
> 
> Все метрики визуализированы в Grafana с красивыми графиками и alerts."

### **Q7: С какими самыми сложными проблемами вы столкнулись и как их решили?**

**Ответ:**
> "Самая сложная проблема была связана с **памятью и стабильностью**.
> 
> **Проблема:**
> Приложение периодически падало. В Grafana я увидел, что память растёт до 100%, затем OOM Killer убивает процессы.
> 
> **Debugging процесс:**
> 1. **Проверил логи:** `docker logs flask-app` показал Out of Memory errors
> 2. **Анализ метрик:** Grafana показала, что память растёт линейно
> 3. **Профилирование:** Добавил memory_profiler в код
> 4. **Обнаружил:** Утечка памяти из-за неправильного кэширования + слишком много Gunicorn workers
> 
> **Решение:**
> 1. **Resource limits в Docker Compose:**
>    ```yaml
>    deploy:
>      resources:
>        limits:
>          memory: 512M
>        reservations:
>          memory: 256M
>    ```
>    Это предотвращает один контейнер от захвата всей памяти
> 
> 2. **Оптимизация Gunicorn:**
>    - Было: 8 workers (слишком много для t2.medium)
>    - Стало: 4 workers + threads
>    - Formula: `(2 x CPU cores) + 1`
> 
> 3. **Redis для session management:**
>    - Вместо хранения сессий в памяти Flask
>    - Redis хранит их отдельно
> 
> 4. **Swap настройка:**
>    - Добавил 2GB swap на сервере
>    - Помогает при временных спайках
> 
> 5. **Database connection pooling:**
>    - SQLAlchemy pool size ограничен
>    - Предотвращает утечку соединений
> 
> 6. **Monitoring alerts:**
>    - Настроил алерт при memory >85%
>    - Получаю уведомление до критической ситуации
> 
> **Результат:**
> - Memory usage стабилизировалась на 60-70%
> - Нет больше OOM crashes
> - Приложение работает стабильно неделями
> 
> **Что я узнал:**
> - Importance of monitoring - без Grafana я бы не нашёл причину
> - Resource limits - ВСЕГДА устанавливайте лимиты в production
> - Profiling - не гадать, а измерять
> - Documentation - я всё задокументировал для будущего
> 
> Эта проблема научила меня больше, чем если бы всё работало сразу."
