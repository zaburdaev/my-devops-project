# Почему меняется IP-адрес и как это исправлено

## Текущий IP-адрес: `18.197.7.122`

---

## 🔄 Почему IP-адрес постоянно менялся?

При каждом запуске пайплайна восстановления инфраструктуры (`infrastructure-recovery.yml`) Terraform **создавал новый Elastic IP** вместо того, чтобы использовать существующий.

### Техническая причина

В Terraform конфигурации Elastic IP был описан как **`resource`** (ресурс):

```hcl
# ❌ Проблемный код — создаёт НОВЫЙ Elastic IP каждый раз
resource "aws_eip" "server_ip" {
  instance = aws_instance.web_server.id
  domain   = "vpc"
}
```

Каждый раз, когда Terraform выполнял `terraform apply`:
1. Старый EC2 инстанс уничтожался (если был)
2. Создавался **новый** EC2 инстанс
3. Создавался **новый** Elastic IP (старый терялся)
4. Новый IP привязывался к новому инстансу

В результате: **каждое восстановление = новый IP-адрес**.

## 🛡️ Как предотвратить смену IP в будущем

Нужно использовать **`data source`** вместо **`resource`** для Elastic IP:

```hcl
# ✅ Правильный подход — использует СУЩЕСТВУЮЩИЙ Elastic IP
data "aws_eip" "server_ip" {
  filter {
    name   = "tag:Name"
    values = ["my-devops-project-eip"]
  }
}

resource "aws_eip_association" "server_ip_assoc" {
  instance_id   = aws_instance.web_server.id
  allocation_id = data.aws_eip.server_ip.id
}
```

### Разница между `resource` и `data source`:

| | `resource` | `data source` |
|---|---|---|
| Что делает | **Создаёт** новый ресурс | **Находит** существующий ресурс |
| При повторном запуске | Удаляет старый, создаёт новый | Использует тот же самый |
| IP-адрес | Меняется каждый раз | Остаётся постоянным |

### Дополнительно: закрепить Elastic IP

Создать Elastic IP **один раз вручную** (или в отдельном Terraform state) и пометить тегом:
```bash
aws ec2 allocate-address --domain vpc --tag-specifications \
  'ResourceType=elastic-ip,Tags=[{Key=Name,Value=my-devops-project-eip}]'
```

После этого пайплайн восстановления будет **находить** этот IP и привязывать к новому инстансу, а не создавать новый.

## 🔧 Что было исправлено в PR #3

**PR #3** добавил в пайплайн восстановления (`infrastructure-recovery.yml`) автоматическое обновление GitHub Secrets после создания инфраструктуры.

### Что именно было добавлено:

```yaml
# Автоматическое обновление секретов GitHub после восстановления
- name: Update GitHub Secrets with new IP
  env:
    GH_TOKEN: ${{ secrets.GH_PAT }}
  run: |
    NEW_IP=$(terraform output -raw elastic_ip)
    gh secret set SERVER_IP --body "$NEW_IP"
    gh secret set ANSIBLE_INVENTORY --body "$NEW_IP,"
```

### Зачем это нужно:

Раньше, после восстановления инфраструктуры с новым IP:
- ❌ CI/CD пайплайн продолжал использовать **старый IP** из GitHub Secrets
- ❌ Деплой приложения **падал**, потому что Ansible не мог подключиться
- ❌ Нужно было **вручную** обновлять секреты в GitHub

Теперь:
- ✅ IP автоматически обновляется в GitHub Secrets
- ✅ CI/CD пайплайн сразу работает с **правильным IP**
- ✅ Деплой проходит **без ручного вмешательства**

## 📋 Рекомендации на будущее

1. **Перевести Elastic IP на `data source`** — чтобы IP не менялся при восстановлении
2. **Создать выделенный Elastic IP** — один раз, вне основного Terraform state
3. **Добавить мониторинг** — оповещение при смене IP-адреса
4. **DNS запись** — использовать доменное имя вместо IP, чтобы достаточно было обновить A-запись
