🔄 Recovery Workflow: EIP Discovery Flow

Автоматическое сохранение IP-адреса при восстановлении инфраструктуры
🔒 Защищённый IP: 18.197.7.122

❌ Старый подход (prevent_destroy)

  • Зависит от terraform.tfstate
  • State файл не доступен на GitHub runner
  • Terraform не знает о существующем EIP
  • Создаёт НОВЫЙ IP адрес каждый раз
  • Результат: IP меняется 💔

✅ Новый подход (EIP Discovery)

  • НЕ зависит от state файла
  • Использует AWS API для поиска EIP
  • Находит EIP по тегу в AWS
  • Передаёт Allocation ID в Terraform
  • Результат: IP СОХРАНЯЕТСЯ ✨

📋 Пошаговый процесс восстановления

1
🚀 Запуск Recovery Workflow
GitHub Actions создаёт новый runner (чистая виртуальная машина без предыдущего state)
Trigger: Manual dispatch / Scheduled
Runner: ubuntu-latest (fresh VM)
State file: ❌ НЕ СУЩЕСТВУЕТ
Terraform state: ❌ ПУСТОЙ
⬇️
2
🔍 Поиск существующего EIP в AWS
Workflow выполняет запрос к AWS API для поиска Elastic IP с тегом "monitoring-eip"
$ aws ec2 describe-addresses \
--filters "Name=tag:Name,Values=monitoring-eip" \
--query 'Addresses[0].AllocationId' \
--output text

✅ Результат: eipalloc-0a5b5e3c8f1d2e4f3
✅ IP адрес: 18.197.7.122
Ключевой момент: EIP хранится в AWS с тегом, который не зависит от Terraform state. Даже если state потерян, мы можем найти правильный EIP!
⬇️
3
📝 Сохранение Allocation ID
Allocation ID сохраняется в GitHub Actions output для использования в следующих шагах
echo "eip_allocation_id=$EXISTING_EIP" >> $GITHUB_OUTPUT

Сохранённая переменная:
eip_allocation_id = "eipalloc-0a5b5e3c8f1d2e4f3"
⬇️
4
🏗️ Terraform Init
Инициализация Terraform без существующего state файла
$ cd terraform
$ terraform init

Initializing the backend...
Terraform has been successfully initialized!
⬇️
5
⚙️ Terraform Apply с EIP переменной
Terraform применяется с передачей Allocation ID существующего EIP через переменную
$ terraform apply -auto-approve \
-var="existing_eip_allocation_id=eipalloc-0a5b5e3c8f1d2e4f3"

Terraform выполняет:
✅ data.aws_eip.existing[0] - Читает существующий EIP
✅ aws_instance.monitoring - Создаёт новый EC2
✅ aws_eip_association.monitoring - Привязывает EIP к EC2
⊘ aws_eip.monitoring[0] - ПРОПУЩЕНО (count = 0)

Apply complete! Resources: 2 added, 0 changed, 0 destroyed.
Магия происходит здесь: Terraform использует условную логику (count) для ПРОПУСКА создания нового EIP и вместо этого использует существующий через data source!
⬇️
6
📄 Обновление документации
Workflow автоматически обновляет README.md с актуальным IP адресом
$ sed -i "s/IP: .*/IP: 18.197.7.122/" README.md
$ git add README.md
$ git commit -m "Update IP address after recovery"
$ git push

✅ Документация обновлена и закоммичена
⬇️
7
🎉 Результат: IP сохранён!
Инфраструктура восстановлена с тем же самым IP адресом

📊 Grafana: http://18.197.7.122:3000
📈 Prometheus: http://18.197.7.122:9090

✅ Все сервисы доступны по прежнему адресу!

🛡️ Гарантии сохранения IP адреса

  • Не зависит от наличия terraform.tfstate файла
  • Работает на любом GitHub Actions runner
  • Использует AWS как единственный источник истины
  • Автоматически находит правильный EIP по тегу
  • Предотвращает создание дубликатов EIP
  • Работает даже если код Terraform изменился
  • Не требует ручного вмешательства

🔧 Ключевые технологии

  • AWS CLI: describe-addresses
  • Terraform: data sources + variables
  • GitHub Actions: outputs между шагами
  • AWS Tags: постоянная идентификация

📦 Terraform компоненты

  • Variable: existing_eip_allocation_id
  • Data Source: aws_eip.existing
  • Resource: aws_eip_association
  • Conditional: count based on variable
