## 📊 Таблица: что удаляется, а что сохраняется

| Ресурс | При Recovery | Причина |
|--------|:----------:|---------|
| 🖥️ EC2 Instance | 🗑️ **Удаляется** | Создаётся заново с чистой ОС |
| 🌐 Elastic IP (18.197.7.122) | ✅ **Сохраняется** | `prevent_destroy = true` в Terraform |
| 🔗 EIP Association | 🗑️ **Удаляется** | Переприкрепляется к новому EC2 |
| 🔐 Security Group | 🗑️ **Удаляется** | Создаётся заново (те же правила) |
| 🔑 Key Pair (Terraform) | 🗑️ **Удаляется** | Генерируется заново |
| 📦 S3 Bucket (tfstate) | ✅ **Сохраняется** | Хранит Terraform state |
| 🗄️ DynamoDB Table | ✅ **Сохраняется** | Хранит Terraform locks |
| 🐳 Docker контейнеры | 🗑️ **Удаляется** | Поднимаются заново Ansible |
| 📊 Данные приложения | 🗑️ **Удаляется** | На новом EC2 чистый диск |

## 🔑 Ключевой механизм: prevent_destroy

```hcl
# terraform/main.tf

resource "aws_eip" "app_ip" {
  domain = "vpc"

  lifecycle {
    prevent_destroy = true    # ← ГЛАВНАЯ ЗАЩИТА!
  }

  tags = {
    Name = "health-dashboard-eip"
  }
}
```

### Как это работает:

```
                    terraform destroy
                          │
          ┌───────────────┼───────────────┐
          ▼               ▼               ▼
    ┌──────────┐   ┌──────────┐   ┌──────────────┐
    │ EC2      │   │ EIP      │   │ Другие       │
    │          │   │          │   │ ресурсы      │
    │ destroy  │   │ BLOCKED! │   │ destroy      │
    │ ✅       │   │ ❌       │   │ ✅           │
    └──────────┘   └──────────┘   └──────────────┘
                        │
                        ▼
              Recovery workflow
              использует selective
              cleanup — удаляет всё
              КРОМЕ EIP из state
```

## 🔄 Детальный поток Recovery Pipeline

```
ЭТАП 1: Selective Cleanup
═══════════════════════════════════════
  terraform state rm aws_eip.app_ip
       │
       ▼  (EIP убран из state, но живёт в AWS)
  terraform destroy -auto-approve
       │
       ▼  (EC2, SG, Association — удалены)
  terraform state list
       │
       ▼  (state пуст)

ЭТАП 2: Import & Rebuild
═══════════════════════════════════════
  terraform import aws_eip.app_ip <eip-alloc-id>
       │
       ▼  (EIP снова в state)
  terraform apply -auto-approve
       │
       ▼  (Новый EC2 + новая Association)
  Результат: тот же IP → новый сервер

ЭТАП 3: Provisioning
═══════════════════════════════════════
  ansible-playbook deploy.yml
       │
       ▼
  SSH → 18.197.7.122 (тот же IP!)
       │
       ▼
  Docker + App установлены
       │
       ▼
  ✅ Сервис доступен на 18.197.7.122:3000
```

## 📝 Примечания

- **Elastic IP** остаётся закреплённым за аккаунтом AWS даже после удаления EC2
- **prevent_destroy** — это страховка от случайного `terraform destroy`
- **Selective cleanup** в pipeline обходит prevent_destroy, удаляя EIP из state перед destroy
- После recovery EIP **импортируется обратно** в state и привязывается к новому серверу
