# 🎬 Сценарий демонстрации проекта

## 📋 Подготовка перед демонстрацией

### За 10 минут до начала:

```bash
# 1. Убедись, что Docker запущен
docker --version

# 2. Перейди в папку проекта
cd my-devops-project

# 3. Запусти все сервисы
docker-compose up -d

# 4. Проверь, что всё работает
docker-compose ps

# 5. Проверь приложение
curl http://localhost:5000/health
```

## 🎤 Типичные вопросы преподавателя и ответы

### «Почему Flask, а не Django?»
> «Flask — легковесный микрофреймворк, идеальный для демонстрации DevOps-процессов. Фокус проекта — на инфраструктуре, а не на сложном backend-коде. Flask простой и понятный.»

### «Что будет, если один из контейнеров упадёт?»
> «Docker Compose перезапустит контейнер благодаря restart policy. В Kubernetes — контроллер автоматически создаст новый pod. Мониторинг покажет alert в Grafana через alert rules в Prometheus.»

### «Можно ли масштабировать приложение?»
> «Да. В Docker Compose: `docker-compose up --scale app=3`. В Kubernetes: изменить `replicaCount` в deployment.yaml или values.yaml. Nginx и Service автоматически балансируют нагрузку.»

### «Зачем Redis, если есть PostgreSQL?»
> «Redis — in-memory хранилище, работает значительно быстрее PostgreSQL для часто запрашиваемых данных. Мы кэшируем метрики на 10 секунд, чтобы не нагружать базу при каждом запросе.»

### «Зачем Nginx перед Flask?»
> «Gunicorn — WSGI-сервер, хорошо обрабатывает Python-код, но не оптимизирован для отдачи статики и обработки HTTPS. Nginx эффективнее работает с HTTP-соединениями, статическими файлами, добавляет security headers и может балансировать нагрузку.»

### «Что такое multi-stage build в Docker?»
> «Это оптимизация размера образа. Первый этап (builder) устанавливает зависимости, второй этап копирует только нужные файлы в чистый образ. В итоге финальный образ значительно меньше.»

### «Как работают health checks?»
> «Приложение имеет endpoint /health, который возвращает JSON со статусом. Docker проверяет этот endpoint каждые 30 секунд. Kubernetes использует liveness probe (перезапуск при сбое) и readiness probe (отключение от балансировщика при неготовности).»

### «Почему деплой в пайплайне не работает?»
> «Деплой-шаг требует настроенного сервера. Без EC2-инстанса (который создаётся через Terraform) и соответствующих GitHub Secrets, этот шаг пропускается. Это сделано намеренно — пайплайн гибкий и не падает из-за отсутствия сервера.»

### «Где хранятся пароли?»
> «Для локальной разработки — в .env файле, который не попадает в Git (.gitignore). Для CI/CD — в GitHub Secrets. Для Kubernetes — в Secret-объектах. В коде только placeholder-значения. Проведён security audit — отчёт есть в проекте.»

### «Можно ли задеплоить в облако?»
> «Да. Шаги: 1) terraform apply — создаёт EC2 сервер, 2) ansible-playbook — настраивает сервер и деплоит приложение, 3) добавить GitHub Secrets — CI/CD начнёт автоматически деплоить. Или развернуть в Kubernetes-кластер через helm install.»
