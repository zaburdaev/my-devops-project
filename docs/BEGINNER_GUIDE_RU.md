## 1. 🌐 Введение в DevOps

### Что такое DevOps?

**DevOps** — это подход к разработке программного обеспечения, который объединяет **разработку** (Development) и **эксплуатацию** (Operations).

Представь себе конвейер на заводе:
- **Разработчик** пишет код (как рабочий собирает деталь)
- **DevOps инженер** строит и обслуживает конвейер, который автоматически проверяет, собирает и доставляет эту деталь заказчику

**Простая аналогия:** Если программист — это повар, который готовит блюдо, то DevOps инженер — это тот, кто построил кухню, наладил поставки продуктов, установил систему контроля качества и организовал доставку блюд клиентам.

### Зачем это нужно?

Без DevOps:
- ❌ Код проверяется вручную → ошибки
- ❌ Деплой делается руками → долго и рискованно
- ❌ Сервера настраиваются вручную → «у меня работало!»
- ❌ Нет мониторинга → узнаём о проблемах от пользователей

С DevOps:
- ✅ Код автоматически тестируется при каждом изменении
- ✅ Деплой происходит автоматически и безопасно
- ✅ Инфраструктура описана как код — воспроизводима
- ✅ Мониторинг показывает проблемы до того, как о них узнает пользователь

### Что делает DevOps инженер?

| Задача | Описание |
|--------|----------|
| CI/CD | Настраивает автоматические пайплайны сборки и деплоя |
| Контейнеризация | Упаковывает приложения в Docker контейнеры |
| Оркестрация | Управляет контейнерами через Kubernetes |
| IaC | Описывает инфраструктуру как код (Terraform) |
| Мониторинг | Настраивает сбор метрик и алертов |
| Автоматизация | Автоматизирует рутинные задачи (Ansible) |
| Безопасность | Следит за безопасностью кода и инфраструктуры |

## 2. 🛠️ Технологии в проекте

### 🐳 Docker (контейнеры)

**Что это?**  
Docker — это инструмент для создания **контейнеров**. Контейнер — это как лёгкая виртуальная машина, которая содержит приложение и всё необходимое для его работы.

**Аналогия:** Представь контейнер для перевозки грузов на корабле. Неважно, что внутри — телевизоры, одежда или еда — контейнер стандартный, его легко грузить и перевозить. Docker делает то же самое с программами.

**Зачем в проекте?**  
- Наше Flask-приложение упаковано в Docker-контейнер
- Это гарантирует, что приложение будет работать одинаково везде: на твоём ноутбуке, на сервере, в облаке

**Как это работает?**  
1. `Dockerfile` — «рецепт» для создания образа (image)
2. Docker читает `Dockerfile` и создаёт образ
3. Из образа запускается контейнер (работающее приложение)

**Где в проекте?**
- `Dockerfile` — описание как собрать образ приложения
- Использует multi-stage build (2 этапа сборки) для уменьшения размера
- Запускается от непривилегированного пользователя (безопасность)

```
Dockerfile
├── Stage 1 (builder) → Установка Python-зависимостей
└── Stage 2 (production) → Лёгкий образ с приложением
```

### 🐳 Docker Compose (оркестрация контейнеров)

**Что это?**  
Docker Compose — инструмент для запуска **нескольких контейнеров** одной командой.

**Аналогия:** Если Docker — это один музыкант, то Docker Compose — это дирижёр оркестра. Он говорит каждому музыканту (контейнеру) когда начинать и как взаимодействовать с другими.

**Зачем в проекте?**  
Наш проект состоит из 7 сервисов, которые должны работать вместе:
- Flask-приложение
- PostgreSQL (база данных)
- Redis (кэш)
- Nginx (веб-сервер)
- Prometheus (метрики)
- Grafana (дашборды)
- Loki (логи)

**Где в проекте?**
- `docker-compose.yml` — описание всех сервисов
- Одна команда `docker-compose up -d` запускает ВСЁ

```yaml
# Упрощённая схема docker-compose.yml:
services:
  app:        # Flask-приложение (порт 5000)
  postgres:   # База данных (порт 5432)
  redis:      # Кэш (порт 6379)
  nginx:      # Веб-сервер (порт 80)
  prometheus: # Метрики (порт 9090)
  grafana:    # Дашборды (порт 3000)
  loki:       # Логи (порт 3100)
```

### 🌶️ Flask (веб-приложение)

**Что это?**  
Flask — это лёгкий фреймворк для создания веб-приложений на Python.

**Аналогия:** Flask — это как конструктор LEGO для веб-сайтов. Простой, гибкий, можно собрать что угодно.

**Зачем в проекте?**  
Это **главное приложение** проекта — Health Dashboard. Оно:
- Показывает дашборд с информацией о системе (CPU, RAM, диск)
- Предоставляет API для проверки здоровья системы
- Отдаёт метрики для Prometheus
- Пишет логи для Loki

**Где в проекте?**
- `app/app.py` — основной код приложения
- `app/wsgi.py` — точка входа для Gunicorn (production-сервер)
- `requirements.txt` — список Python-зависимостей

**Основные эндпоинты (URL-адреса):**

| URL | Что делает |
|-----|-----------|
| `/` | Главная страница — HTML-дашборд |
| `/health` | Проверка здоровья приложения (JSON) |
| `/metrics` | Метрики для Prometheus |
| `/api/system-info` | Информация о системе (CPU, RAM, диск) |

### 🐘 PostgreSQL (база данных)

**Что это?**  
PostgreSQL — это реляционная база данных. Она хранит данные в таблицах (как Excel, но мощнее).

**Аналогия:** Если приложение — это магазин, то PostgreSQL — это склад, где хранятся все товары (данные).

**Зачем в проекте?**  
- Хранит исторические метрики системы (CPU, RAM, диск)
- Приложение записывает данные при каждом сборе метрик

**Где в проекте?**
- Запускается как контейнер через `docker-compose.yml`
- Данные сохраняются в Docker volume `postgres_data`
- Настройки подключения — в переменных окружения (`.env`)

### ⚡ Redis (кэш)

**Что это?**  
Redis — это хранилище данных в оперативной памяти. Работает очень быстро.

**Аналогия:** Redis — это как записная книжка на рабочем столе. Она маленькая, но обращаться к ней намного быстрее, чем идти на склад (базу данных).

**Зачем в проекте?**  
- Кэширует результаты системных метрик на 10 секунд
- Уменьшает нагрузку на базу данных
- Ускоряет ответы API

**Где в проекте?**
- Запускается как контейнер через `docker-compose.yml`
- Данные сохраняются в Docker volume `redis_data`

### 🌐 Nginx (веб-сервер / reverse proxy)

**Что это?**  
Nginx — это высокопроизводительный веб-сервер, который работает как «входная дверь» для приложения.

**Аналогия:** Nginx — это как администратор на ресепшене в офисе. Он встречает всех посетителей (запросы) и направляет их к нужному сотруднику (сервису).

**Зачем в проекте?**  
- Принимает HTTP-запросы на порт 80
- Перенаправляет их к Flask-приложению на порт 5000
- Добавляет заголовки безопасности
- Может балансировать нагрузку между несколькими копиями приложения

**Где в проекте?**
- `nginx/nginx.conf` — конфигурация Nginx
- Запускается как контейнер через `docker-compose.yml`

### 🏗️ Terraform (инфраструктура как код)

**Что это?**  
Terraform — инструмент для создания облачной инфраструктуры с помощью кода.

**Аналогия:** Вместо того чтобы вручную кликать кнопки в AWS-консоли, ты описываешь инфраструктуру в файлах. Это как чертёж для строительства — по нему можно построить одинаковый дом сколько угодно раз.

**Зачем в проекте?**  
- Создаёт EC2-сервер в AWS
- Настраивает Security Group (правила файрвола)
- Автоматически устанавливает Docker на сервер

**Где в проекте?**
- `terraform/main.tf` — основная конфигурация (EC2, Security Group)
- `terraform/variables.tf` — переменные (регион, тип сервера, SSH-ключ)
- `terraform/outputs.tf` — выходные данные (IP-адрес сервера, URL)

**Основные команды:**
```bash
terraform init    # Инициализация (скачать провайдеры)
terraform plan    # Показать что будет создано
terraform apply   # Создать инфраструктуру
terraform destroy # Удалить всё
```

### ☸️ Kubernetes (оркестрация контейнеров)

**Что это?**  
Kubernetes (K8s) — система для автоматического управления контейнерами в масштабе.

**Аналогия:** Если Docker Compose — это дирижёр маленького оркестра, то Kubernetes — это менеджер огромной филармонии. Он управляет сотнями музыкантов, заменяет заболевших, добавляет новых при необходимости.

**Зачем в проекте?**  
- Показывает как приложение можно развернуть в продакшен
- Автоматический перезапуск упавших контейнеров
- Масштабирование (можно легко добавить копии приложения)
- Балансировка нагрузки

**Где в проекте?**
- `k8s/namespace.yaml` — пространство имён
- `k8s/configmap.yaml` — конфигурация (не секретная)
- `k8s/secret.yaml` — секреты (пароли)
- `k8s/deployment.yaml` — описание деплоймента (2 реплики)
- `k8s/service.yaml` — сервис (LoadBalancer)
- `k8s/helm/` — Helm-чарт (шаблонизация манифестов)

### 🤖 Ansible (автоматизация)

**Что это?**  
Ansible — инструмент для автоматической настройки серверов.

**Аналогия:** Ansible — это как подробная инструкция для нового сотрудника. Вместо того чтобы каждый раз объяснять, как настроить рабочее место, ты даёшь ему чек-лист, и он всё делает сам.

**Зачем в проекте?**  
- Автоматически устанавливает Docker на сервер
- Копирует файлы проекта на сервер
- Запускает приложение через Docker Compose
- Настраивает файрвол

**Где в проекте?**
- `ansible/inventory.ini` — список серверов
- `ansible/playbook.yml` — основной плейбук (что делать)
- `ansible/roles/docker/` — роль для установки Docker
- `ansible/roles/app/` — роль для деплоя приложения

### ⚙️ GitHub Actions (CI/CD)

**Что это?**  
GitHub Actions — это сервис автоматизации прямо в GitHub. CI/CD означает:
- **CI** (Continuous Integration) — автоматическая проверка кода
- **CD** (Continuous Delivery/Deployment) — автоматическая доставка кода

**Аналогия:** Представь, что каждый раз, когда ты кладёшь письмо в почтовый ящик (push в GitHub), автоматически:
1. Робот проверяет, нет ли ошибок в письме (тесты)
2. Другой робот красиво его упаковывает (сборка Docker-образа)
3. Третий робот доставляет его получателю (деплой на сервер)

**Зачем в проекте?**  
- Автоматически запускает тесты при каждом push
- Собирает Docker-образ и загружает его на Docker Hub
- Деплоит приложение на сервер (если настроен)

**Где в проекте?**
- `.github/workflows/ci-cd.yml` — описание пайплайна

**Структура пайплайна:**
```
git push → [Тесты ✅] → [Сборка Docker 🐳] → [Деплой 🚀]
                ↓               ↓                   ↓
          pytest + flake8   Push на Docker Hub   SSH на сервер
```

### 📊 Prometheus (метрики)

**Что это?**  
Prometheus — система мониторинга, которая собирает числовые метрики с приложений.

**Аналогия:** Prometheus — это как датчики в автомобиле. Они постоянно показывают скорость, температуру двигателя, уровень топлива. Если что-то не так — загорается лампочка.

**Зачем в проекте?**  
- Каждые 10 секунд запрашивает метрики у приложения (`/metrics`)
- Хранит историю метрик (CPU, RAM, диск, количество запросов)
- Позволяет настроить алерты (оповещения)

**Где в проекте?**
- `monitoring/prometheus.yml` — конфигурация Prometheus
- `monitoring/alert_rules.yml` — правила алертов
- Доступен по адресу: `http://localhost:9090`

### 📈 Grafana (визуализация)

**Что это?**  
Grafana — инструмент для создания красивых дашбордов и графиков.

**Аналогия:** Если Prometheus — это датчики, то Grafana — это приборная панель автомобиля, где ты видишь все показатели в удобном виде.

**Зачем в проекте?**  
- Визуализирует метрики из Prometheus
- Показывает графики CPU, RAM, диска
- Показывает логи из Loki
- Готовый дашборд «Health Dashboard» уже настроен

**Где в проекте?**
- `grafana/provisioning/` — автоматическая настройка
- `grafana/provisioning/dashboards/health-dashboard.json` — готовый дашборд
- Доступна по адресу: `http://localhost:3000` (admin/admin)

---

### 📝 Loki (логи)

**Что это?**  
Loki — система для сбора и хранения логов (записей о работе приложения).

**Аналогия:** Loki — это как бортовой журнал самолёта. Записывает всё, что происходит. Если что-то пошло не так — можно открыть журнал и разобраться.

**Зачем в проекте?**  
- Собирает JSON-логи от Flask-приложения
- Позволяет искать и фильтровать логи в Grafana
- Хранит историю событий

**Где в проекте?**
- `monitoring/loki-config.yaml` — конфигурация Loki
- Доступен по адресу: `http://localhost:3100`
- Логи просматриваются через Grafana → Explore → Loki

## 4. ⚙️ Как это работает

### 4.1 Что происходит, когда ты делаешь `git push`

```
Ты: git push origin main
        │
        ▼
┌─── GitHub получает код ───┐
│                            │
│  GitHub Actions срабатывает │
│  (.github/workflows/       │
│   ci-cd.yml)               │
│                            │
└────────┬───────────────────┘
         │
         ▼
┌─── Job 1: ТЕСТЫ ───────────┐
│                              │
│  1. Скачивает код            │
│  2. Ставит Python 3.11       │
│  3. Ставит зависимости (pip) │
│  4. Запускает pytest         │
│  5. Запускает flake8 (линт)  │
│                              │
│  ✅ Тесты прошли!            │
└────────┬─────────────────────┘
         │
         ▼
┌─── Job 2: СБОРКА ───────────┐
│                              │
│  1. Логинится в Docker Hub   │
│  2. Собирает Docker-образ    │
```

## 7. ❓ Частые вопросы и ответы

### Docker не запускается

**Проблема:** Docker Desktop не стартует или показывает ошибку.

**Решения:**
```bash
# Windows: Проверь, что WSL2 включён
wsl --install
wsl --update

# Перезапусти Docker Desktop
# Правый клик на иконку Docker → Restart

# Linux: Проверь статус сервиса
sudo systemctl status docker
sudo systemctl start docker
```

### Порты заняты

**Проблема:** `Error: port is already allocated`

**Решения:**
```bash
# Посмотри, что занимает порт (например, 5000)
# Windows:
netstat -ano | findstr :5000

# Linux/Mac:
lsof -i :5000

# Останови другой процесс или измени порт в docker-compose.yml
# Например: ports: "5001:5000" вместо "5000:5000"
```

### Тесты падают

**Проблема:** `pytest` показывает ошибки.

**Решения:**
```bash
# Убедись, что зависимости установлены
pip install -r requirements.txt

# Запусти тесты с подробным выводом
python -m pytest tests/ -v --tb=long

# Тесты не требуют БД и Redis — они используют Flask test client
```

### Как посмотреть логи контейнеров

```bash
# Все логи
docker-compose logs

# Логи конкретного сервиса
docker-compose logs app
docker-compose logs postgres
docker-compose logs nginx

# Логи в реальном времени (Ctrl+C для выхода)
docker-compose logs -f

# Последние 50 строк
docker-compose logs --tail=50 app
```

### Как остановить все контейнеры

```bash
# Остановить (данные сохраняются)
docker-compose down

# Остановить и удалить данные (volumes)
docker-compose down -v

# Или через Makefile
make down
```

### Как пересобрать образы

```bash
# Пересобрать без кэша
docker-compose build --no-cache

# Пересобрать и запустить
docker-compose up -d --build

# Или через Makefile
make build
make up
```

### Grafana показывает «No data»

**Решения:**
1. Подожди 1-2 минуты — Prometheus нужно время для сбора метрик
2. Проверь, что приложение работает: `curl http://localhost:5000/health`
3. Проверь Prometheus: http://localhost:9090 → Status → Targets → все должны быть UP
4. В Grafana проверь data source: Configuration → Data Sources → Prometheus → Test

### Docker Compose показывает ошибку при запуске

```bash
# Проверь .env файл
cat .env

# Если .env нет:
cp .env.example .env

# Проверь синтаксис docker-compose.yml
docker-compose config

# Попробуй с пересборкой
docker-compose up -d --build
```

### Как отвечать на вопросы

**«Что такое Docker?»**
> «Docker — инструмент контейнеризации. Контейнер содержит приложение и все зависимости, что обеспечивает одинаковую работу в любой среде.»

**«Зачем нужен CI/CD?»**
> «CI/CD автоматизирует тестирование, сборку и деплой. При каждом push код автоматически проверяется и собирается Docker-образ.»

**«Зачем нужен Kubernetes, если есть Docker Compose?»**
> «Docker Compose подходит для разработки и небольших проектов. Kubernetes — для production: автоматическое масштабирование, self-healing, балансировка нагрузки.»

**«Зачем Terraform, если можно создать сервер руками?»**
> «Terraform — Infrastructure as Code. Код можно версионировать, воспроизводить и автоматизировать. Один `terraform apply` создаёт всю инфраструктуру.»

**«Зачем Ansible, если есть Terraform?»**
> «Terraform создаёт инфраструктуру (сервер), Ansible настраивает её (устанавливает ПО, деплоит приложение). Они дополняют друг друга.»

**«Как работает мониторинг?»**
> «Prometheus каждые 10 секунд собирает метрики с приложения через endpoint /metrics. Grafana визуализирует эти метрики в виде графиков. Loki собирает логи.»

**«Почему используется multi-stage build в Docker?»**
> «Multi-stage build уменьшает размер финального образа. Первый этап устанавливает зависимости, второй копирует только нужное в чистый образ.»

**«Что будет, если один контейнер упадёт?»**
> «В Docker Compose — контейнер перезапустится (restart policy). В Kubernetes — система автоматически заменит упавший pod.»
