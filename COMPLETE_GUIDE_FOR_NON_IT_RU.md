## 1. ВВЕДЕНИЕ - ЧТО МЫ ВООБЩЕ СДЕЛАЛИ?

Если совсем просто, мы собрали «умный сайт» для контроля состояния системы.

Что именно сделали:

- Создали веб-приложение (по сути как обычный сайт).
- Сделали автоматическую проверку «здоровья» приложения (работает ли, не перегружено ли, есть ли ошибки).
- Разместили всё на удалённом компьютере в облаке (не у вас дома, а в дата-центре Amazon).
- Настроили процесс так, чтобы можно было быстро восстановить систему после поломки.

### Простая картина мира

Представьте автомобиль с автопилотом:

- Водитель = разработчик
- Автомобиль = наше приложение
- Приборная панель = мониторинг (Grafana)
- Автосервис и техосмотр = тесты и CI/CD
- Запасной план при поломке = recovery/runbook (восстановление)

### Важные аналогии

- **GitHub** — это как Google Drive для программистов, но с историей всех изменений и возможностью откатиться назад.
- **Docker** — это как контейнер для перевозки: внутри уже всё упаковано, и груз доедет одинаково в любую точку.
- **Облако AWS** — это как аренда удалённого компьютера: не покупаете «железо», а берёте в аренду.

## 2. ОСНОВНЫЕ КОМПОНЕНТЫ - ИЗ ЧЕГО СОСТОИТ ПРОЕКТ

### 2.1 Приложение (Flask App)

**Что такое Flask?**

Flask — это «набор готовых деталей» для создания веб-приложений на Python.

- Не нужно писать всё с нуля.
- Есть готовые механизмы для страниц, API, ответов браузеру.

**Что делает наше приложение?**

- Отдаёт страницу/данные о состоянии системы.
- Имеет endpoint `/health`, который отвечает, что сервис жив.
- Отдаёт метрики для мониторинга.

**Как открыть?**

- Локально: `http://localhost:5000/health`
- На сервере: `http://ВАШ_IP:5000/health`

**Аналогия:**

Это как корпус автомобиля и панель управления — видимая часть для пользователя.

### 2.2 База данных (PostgreSQL)

**Зачем нужна база?**

Чтобы хранить данные надёжно и надолго.

**Что хранит у нас?**

В учебном/демо-проекте может хранить служебную информацию, историю, настройки, данные для API.

**Аналогия:**

Это как память в виде записной книжки: всё важное записывается и не теряется после перезапуска.

### 2.3 Кэш (Redis)

**Что такое кэш?**

Кэш — это очень быстрая временная память для часто используемых данных.

**Зачем нужен?**

- Чтобы не обращаться каждый раз в медленное хранилище.
- Чтобы ускорить ответы приложения.

**Аналогия:**

Это как держать часто нужные вещи на столе, а не каждый раз идти в кладовку.

### 2.4 Веб-сервер (Nginx)

**Что делает Nginx?**

- Принимает запросы от пользователей.
- Передаёт их во Flask.
- Может добавлять защитные заголовки и балансировать нагрузку.

**Аналогия:**

Это как секретарь: принимает звонок и направляет к нужному сотруднику.

### 2.5 Мониторинг (Prometheus + Grafana)

**Prometheus**:

- Регулярно собирает показатели (CPU, память, статус, время ответа).
- Как медицинские датчики.

**Grafana**:

- Показывает эти данные в виде понятных графиков и панелей.

**Аналогия:**

Это как приборная панель автомобиля: видите скорость, температуру, предупреждения.

## 3. ТЕХНОЛОГИИ - ИНСТРУМЕНТЫ КОТОРЫЕ МЫ ИСПОЛЬЗОВАЛИ

### 3.1 Docker (Контейнеризация)

**Что такое контейнер?**

Это «упаковка» приложения вместе со всем, что ему нужно для запуска (библиотеки, зависимости, настройки).

**Зачем это нужно?**

- На любом компьютере запускается одинаково.
- Уменьшается риск ошибки «у меня работает, у тебя — нет».

**Преимущества:**

- Повторяемость
- Быстрый запуск
- Простое масштабирование

**Аналогия:**

Представьте коробку IKEA: внутри всё нужное для сборки, и результат всегда одинаковый.

### 3.2 GitHub (Хранилище кода)

**Что такое Git?**

Git — это система контроля версий. Она хранит историю изменений по шагам.

**Зачем GitHub?**

GitHub — онлайн-платформа для Git:

- хранит код в облаке,
- запускает автоматические проверки,
- помогает работать командой.

**Как работает?**

Вы сохраняете изменения локально (`commit`) и отправляете в GitHub (`push`).

**Аналогия:**

Google Drive + история изменений + автоматическая проверка качества.

### 3.3 AWS (Облако)

**Что такое облако?**

Удалённые серверы, которые вы арендуете.

**EC2**:

Виртуальный компьютер, где работает ваше приложение.

**Elastic IP**:

Постоянный публичный адрес сервера, чтобы ссылка не менялась.

**Аналогия:**

Вместо покупки дорогого сервера вы арендуете его у Amazon как сервис.

### 3.4 Terraform (Инфраструктура как код)

**Что значит «инфраструктура как код»?**

Серверы и сеть описываются в файле, а не вручную через кнопки.

**Зачем?**

- Быстро повторить окружение.
- Меньше человеческих ошибок.
- Легче восстановиться после аварии.

**Аналогия:**

Это как точный чертёж дома: по нему можно построить снова 1 в 1.

### 3.5 Ansible (Автоматизация)

**Что делает Ansible?**

Автоматически настраивает сервер:

- устанавливает нужные программы,
- копирует файлы,
- запускает сервисы.

**Чем отличается от Terraform?**

- Terraform создаёт «дом» (инфраструктуру).
- Ansible обустраивает «дом» внутри (приложение и окружение).

**Аналогия:**

Terraform строит дом, Ansible расставляет мебель и включает технику.

### 3.6 CI/CD (Автоматическая доставка)

**CI (Continuous Integration)**:

Автоматическая проверка нового кода (тесты, сборка).

**CD (Continuous Deployment/Delivery)**:

Автоматическая доставка на сервер после успешных проверок.

**Как работает?**

Сделали push в GitHub → всё запускается автоматически.

**Аналогия:**

Это как конвейер на заводе: деталь поступила — её проверили, собрали, отправили.

## 4. КАК ВСЁ ЭТО РАБОТАЕТ ВМЕСТЕ?

### 4.1 Жизненный цикл изменения

#### Шаг 1: Программист меняет код

```text
Ты пишешь новую функцию в файле app.py на своём компьютере
```

#### Шаг 2: Сохранение в Git

```bash
git add .
git commit -m "Добавил новую функцию"
git push
```

Объяснение: это как сохранить документ и отправить его на «Google Drive для кода».

#### Шаг 3: GitHub Actions запускается автоматически

```text
GitHub видит изменения и запускает pipeline
```

#### Шаг 4: Автоматические тесты

```text
Проверяется, что код работает корректно (например, 12 тестов).
Если хотя бы один тест не прошёл — процесс останавливается.
```

Аналогия: как техосмотр автомобиля перед выездом.

#### Шаг 5: Сборка Docker-образа

```text
Код упаковывается в Docker-образ.
Образ отправляется в Docker Hub (склад контейнеров).
```

#### Шаг 6: Деплой на сервер

```text
Сервер получает команду:
- Скачать свежий код
- Запустить/перезапустить контейнеры
- Проверить что сервис поднялся
```

#### Шаг 7: Проверка работоспособности

```text
Grafana показывает, что приложение живо и метрики обновляются
```

### 4.2 Схема взаимодействия компонентов

```text
Пользователь → Nginx → Flask App → PostgreSQL/Redis
                                  ↓
                              Prometheus → Grafana
```

**Расшифровка стрелок простыми словами:**

1. **Пользователь → Nginx**
   - Человек открывает сайт, запрос сначала приходит в Nginx.
2. **Nginx → Flask App**
   - Nginx передаёт запрос в приложение.
3. **Flask App → PostgreSQL/Redis**
   - Приложение читает/пишет данные в базу и использует кэш для ускорения.
4. **Flask App → Prometheus**
   - Приложение отдаёт метрики о своём состоянии.
5. **Prometheus → Grafana**
   - Prometheus собирает цифры, Grafana рисует понятные графики.

### 5.4 ШАГ 3: НАПИСАНИЕ КОДА ПРИЛОЖЕНИЯ

Создайте файл `app/app.py`:

```python
from flask import Flask

app = Flask(__name__)

@app.route('/health')
def health():
    return {"status": "healthy"}

if __name__ == '__main__':
    app.run(host='0.0.0.0', port=5000)
```

#### По строкам

- `from flask import Flask` — подключаем инструмент Flask.
- `app = Flask(__name__)` — создаём приложение.
- `@app.route('/health')` — делаем страницу/endpoint `/health`.
- `return {"status": "healthy"}` — отдаём ответ.
- `app.run(...)` — запускаем сервер приложения.

### 5.5 ШАГ 4: СОЗДАНИЕ DOCKERFILE

Создайте файл `Dockerfile`:

```dockerfile
FROM python:3.11
WORKDIR /app
COPY requirements.txt .
RUN pip install -r requirements.txt
COPY . .
CMD ["python", "app.py"]
```

#### По строкам

- `FROM python:3.11` — базовый образ с Python.
- `WORKDIR /app` — рабочая папка внутри контейнера.
- `COPY requirements.txt .` — копируем список зависимостей.
- `RUN pip install -r requirements.txt` — устанавливаем зависимости.
- `COPY . .` — копируем код проекта.
- `CMD ...` — команда запуска контейнера.

### 5.6 ШАГ 5: DOCKER COMPOSE

`docker-compose.yml` — это «дирижёр оркестра», который запускает сразу несколько сервисов вместе.

Пример структуры:

```yaml
services:
  app:
    build: .
    ports:
      - "5000:5000"
  postgres:
    image: postgres:15
  redis:
    image: redis:7
  nginx:
    image: nginx:alpine
  prometheus:
    image: prom/prometheus
  grafana:
    image: grafana/grafana
```

Что это значит:

- `app` — ваше приложение.
- `postgres` — база данных.
- `redis` — быстрый кэш.
- `nginx` — входная точка запросов.
- `prometheus` — сборщик метрик.
- `grafana` — красивые графики и дашборды.

### 5.7 ШАГ 6: НАСТРОЙКА TERRAFORM

#### Что делает Terraform

- Создаёт виртуальный сервер EC2.
- Настраивает правила доступа (security group).
- Создаёт/привязывает Elastic IP.

#### Пошагово

1. Создайте AWS Access Key + Secret Key.
2. Создайте `terraform/main.tf`.
3. Выполните:

```bash
cd terraform
terraform init

terraform apply
```

- `init` — подготовка Terraform.
- `plan` — предпросмотр изменений.
- `apply` — применить и создать ресурсы.

### 5.8 ШАГ 7: НАСТРОЙКА CI/CD

Создайте `.github/workflows/ci-cd.yml`.

Этапы:

1. **Trigger** — запуск при `push` в `main`.
2. **Test** — прогон тестов.
3. **Build** — сборка Docker-образа.
4. **Deploy** — выкладка на сервер.

Мини-логика:

- Если тесты не прошли → деплой не выполняется.
- Если всё прошло → сервер обновляется автоматически.

### 5.11 ШАГ 10: НАСТРОЙКА GITHUB ACTIONS

#### Добавить секреты в GitHub

Открыть: `Repository → Settings → Secrets and variables → Actions`.

Добавить:

- `AWS_ACCESS_KEY_ID`
- `AWS_SECRET_ACCESS_KEY`
- `DOCKER_USERNAME`
- `DOCKER_PASSWORD` (или token)
- `SERVER_HOST`
- `SERVER_USER`
- `SSH_PRIVATE_KEY`

#### Почему секреты нельзя хранить в коде?

Потому что код часто публичный или доступен команде. Пароли в коде = очень высокий риск утечки.

## 6. ЧАСТЫЕ ВОПРОСЫ (FAQ)

### Что делать если приложение не запускается?

Проверка по шагам:

1. Посмотреть контейнеры:
   ```bash
   docker compose ps
   ```
2. Посмотреть логи:
   ```bash
   docker compose logs --tail=100 app
   ```
3. Проверить `.env` (пароли, хосты, порты).
4. Перезапустить:
   ```bash
   docker compose down
   docker compose up -d --build
   ```

### Что делать если GitHub Actions падает?

1. Открыть вкладку **Actions**.
2. Найти красный шаг (ошибка).
3. Прочитать точную строку ошибки.
4. Исправить причину и сделать новый `git push`.

#### Пример типичной ошибки №1 (по скриншоту)

**Описание скриншота `IMG_20260413_002753_0.jpg`:**
- Ошибка: `cd: /opt/health-dashboard: No such file or directory`
- Смысл: на сервере не существует нужная папка.
- Решение: в deploy-скрипте добавить шаг «если папки нет — сделать `git clone`».

#### Пример типичной ошибки №2 (по скриншоту)

**Описание скриншота `IMG_20260413_003648_0.jpg`:**
- Ошибка: `sudo: git: command not found`
- Ошибка: `cp: cannot stat '.env.example'`
- Ошибка: `no configuration file provided: not found`
- Смысл: на сервере не установлены нужные инструменты/нет нужных файлов.
- Решение: установить `git`, проверить что репозиторий корректно склонирован, убедиться что `docker-compose.yml` и `.env.example` существуют.

### Что делать если Grafana не показывает данные?

1. Проверить, запущен ли Prometheus.
2. Проверить datasource в Grafana (URL: `http://prometheus:9090`).
3. Проверить endpoint метрик приложения `/metrics`.
4. Проверить, что scrape job в `prometheus.yml` правильный.

### Что делать если сервер недоступен?

1. Проверить статус EC2 в AWS.
2. Проверить Security Group (открыты нужные порты).
3. Проверить, не изменился ли Elastic IP.
4. При необходимости выполнить recovery runbook (Terraform + Ansible + обновить GitHub Secrets).

### Как обновить приложение?

1. Изменить код.
2. `git add`, `git commit`, `git push`.
3. Дождаться завершения GitHub Actions.
4. Проверить endpoint `/health` и Grafana.

### Как добавить новую функцию? (пример `/status`)

1. Добавить новый route в Flask:

```python
@app.route('/status')
def status():
    return {"service": "ok", "version": "1.0"}
```

2. Добавить тест.
3. Сделать commit/push.
4. Проверить после деплоя.

### Как изменить пароли?

- Локально: изменить в `.env` и перезапустить сервисы.
- В CI/CD: изменить в GitHub Secrets.
- На сервере: обновить `.env` и `docker compose up -d`.

### Как настроить HTTPS?

Базовый путь:

1. Зарегистрировать домен.
2. Направить домен на ваш сервер (DNS A record).
3. Установить Certbot.
4. Выпустить сертификат Let’s Encrypt.
5. Настроить Nginx на `443`.

- **API** — способ, как программы «разговаривают» друг с другом.
- **Backend** — серверная часть (невидимая пользователю логика).
- **Frontend** — видимая часть, интерфейс.
- **Container** — упакованная программа со всем необходимым.
- **Deploy** — выкладка приложения на сервер.
- **Repository (Repo)** — папка проекта с историей изменений.
- **Commit** — сохранённый шаг изменений в Git.
- **Push** — отправка изменений в удалённый репозиторий.
- **Pipeline** — цепочка автоматических шагов (тест, сборка, деплой).
- **Monitoring** — наблюдение за состоянием системы.
- **Metric** — измеримый показатель (CPU, память, latency).
- **Healthcheck** — проверка «жив ли сервис».
- **Infrastructure as Code** — управление инфраструктурой через файлы кода.
